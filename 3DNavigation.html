Custom 3D NavMesh

Partitioning space with Octree datastructure and finding a quick path between cells with A* algorithm.

Project goal : Build a dynamic navigation mesh for flying agents in 3 dimentional space.

The idea of this project was born during a Game Jam where the player was fighting flying enemies, and their pathfinding wasn't as fluent as I wanted it to be.

After reading a few things about navigation, I've thought that the octree datastructure was an efficient way to retrace scene geometry.

Subdividing occupied nodes is adding complexity only where it's required to have a precise representation of obstacles.

The OctreeNode class is a simple C# class, containing data like position or depth, and a few methods like the recursive Split.

The Navigation Volume manages the Root Node, the visual representation and the navmesh actualization.

Tolerance is the purcentage of the node that must be occupied to subdivide.

The next step was to find an alogrithm to find neighbors Nodes.

I first tried a system of adresses, with each nodes having a binary adress.

But I was stuck, so I've decided to change my paradigm and to use geometry and positions calculations to find the closest nodes.

With this new approach, I was quickly able to find neighbors Nodes and to have this new Graph, called a Dual.

I'm not yet satisfied by the cost of the algorithm which remains high, so I'll keep optimizing it in the future.

On the left, you can see every leaf Node relied to her neighbors.

Those neighbors doesn't need to be found at initialization, but I've kept a debug mode were the dual is build because it looks really pretty to me. And for debugging. Of course.

The next step was to implement A*.

I've implemented the basic version of A*, and now the Agent is able to move to his destination.

The path can be quickly recalculated at runtime if the destination of the agent changes.

The path is still suboptimal, and my next goal is to add curves to the motion make the movement more fluid.
