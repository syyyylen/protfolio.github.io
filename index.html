Custom 3D NavMesh

Partitioning space with Octree datastructure and finding a quick path between cells with A* algorithm.

Project goal : Build a dynamic navigation mesh for flying agents in 3 dimentional space.

The idea of this project was born during a Game Jam where the player was fighting flying enemies, and their pathfinding wasn't as fluent as I wanted it to be.

After reading a few things about navigation, I've thought that the octree datastructure was an efficient way to retrace scene geometry.

Subdividing occupied nodes is adding complexity only where it's required to have a precise representation of obstacles.

The OctreeNode class is a simple C# class, containing data like position or depth, and a few methods like the recursive Split.

The Navigation Volume manages the Root Node, the visual representation and the navmesh actualization.

Tolerance is the purcentage of the node that must be occupied to subdivide.

The next step was to find an alogrithm to find neighbors Nodes.

I first tried a system of adresses, with each nodes having a binary adress.

But I was stuck, so I've decided to change my paradigm and to use geometry and positions calculations to find the closest nodes.

With this new approach, I was quickly able to find neighbors Nodes and to have this new Graph, called a Dual.

I'm not yet satisfied by the cost of the algorithm which remains high, so I'll keep optimizing it in the future.

On the left, you can see every leaf Node relied to her neighbors.

Those neighbors doesn't need to be found at initialization, but I've kept a debug mode were the dual is build because it looks really pretty to me. And for debugging. Of course.

The next step was to implement A*.

I've implemented the basic version of A*, and now the Agent is able to move to his destination.

The path can be quickly recalculated at runtime if the destination of the agent changes.

The path is still suboptimal, and my next goal is to add curves to the motion make the movement more fluid.

Before Dusk

Last year school project, 8 months of production.
A Voodoo doll explores a jungle using her power to transfer everything harmful that happens to her.
Strong needs in code architecture, optimization and programming physics.

In Before Dusk, this charming Voodoo Doll escapes slavery and crosses a jungle, seeking for the sorceress that imprisoned her soul.

The player embodies this revolted doll, which has the ability to create a Voodoo link with objects or living things.

When she is linked to something, everything bad or harmful that happens to her will equally be transfered to the linked entity.

One of the main challenges in terms of code on this project was the sandbox aspect of the in game interactions.

In the game, the player can catch fire, be thrown away by an explosion, hit walls severely... And all those things must happen simultaneously to the linked object or opponent.

We based our architecture on an Interface, IEffectable, which guarantees that the object can take effects.
Effects can be triggered under conditions or transfered by the player, which is also an IEffectable.

To help the Level Design, avoid weird behaviors and game breaking interactions we needed a custom Editor for IEffectables objects.

In the editor, we simulate behaviors to anticipate what will happen in game (fire propagation, explosion on the right).

Physics interactions aren't handled by Unity's physics engine.
We take control over displacements to have predictible movements which fit our metrics and sometimes unrealistic behavior that serves the gameplay.

Boss Fight in Unreal Engine

My goal was to practice AI and learn about state machines and behavior trees.
In this project, the Bear is a Boss, chasing and attacking player with both range and melee attack.

His behavior is based on Unreal Blackboard and Behavior Tree.

Transitions between states are triggered by conditions. When fullfiled, the bear switch state. Those conditions are easier to fullfill when the Boss his low on health, as a fury mode.

Contained

Game jam game, made in 48 hours with a friend of mine on Unity.
The player fights enemy waves of drones.
FPS Character Controller, flying drones and wave management.

Contained core gameplay is fairly simple : shoot at enemy waves.

This Game Jam was only about having fun for us, and FPS character controller was a great coding experience.
Player movements allows a lot of mobility : fast movements and double jump ability.

To iterate quickly on balancing, the Wave Manager was made to create new Waves quickly, using Unity data assets : Scriptable Objects.

Flying enemy drones are trying to reach anoptimal orbit range before rotating around player, and are attacking him with a slight offset.
